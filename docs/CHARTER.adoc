== Identity

IS::
A standalone, per-repo CI/CD fighter that drives a stuck pipeline to a *legitimate*
green, or improves red→yellow, or fails and emits a structured human report.

IS-NOT::
* A bypass. Never `enforce_admins=off`, never "drop the required context", never
  admin-merge. It *wins* the gate; it does not cheat past it.
* hypatia/estate-dependent. Detachable is the whole point — the genesis is people
  who admire the big system but want an intelligent local CI/CD fighter without
  adopting hypatia.

== Core principle: squabble ≠ bypass

Every "green" the squabbler declares must be backed by the gate's *own* required
check passing legitimately — never by an admin override, an `enforce_admins`
toggle, or removal/rename-away of a required context. An admin-merge cheats past
the gate; a squabble *wins* it.

== Architecture (3 Rust crates, one repo)

`squabble-core`::
The fight engine. Pure, detachable, the valuable reusable part. Loop:
`diagnose → propose move → apply → verify-satisfied`.

`squabble-cli` (`squabble` bin)::
Wraps core. Runs locally or as a CI step. git + `gh` plumbing.

`squabble-app`::
GitHub App — axum webhook server + octocrab. Delivers the trigger button; calls
the same `squabble-core`. *Provisional:* hosted on the farm (owned compute, per
the cost-governor doctrine), not a conventional hosted App — owner to confirm.

== The SPARK formal heart

Model the gate as a state machine and *prove in SPARK that the only transition
into `GREEN` is "a required check actually ran and passed."* Transitions such as
"required context removed", "admin override", "check renamed away" are provably
*unreachable* as paths to green. This makes `squabble ≠ bypass` a machine-checked
invariant rather than a code-review promise — a clean, bounded SPARK target.

*Provisional:* ships *in* v0.1 (the principle cannot be a v0.2 afterthought) —
owner to confirm.

== The arena

The user pre-selects the permitted landing strategies. Estate default =
`{squash, rebase}` + auto-variants, *never* merge-commit (linear-history
rulesets). The squabbler lands only via a permitted strategy, or it does not land.

== Trigger surface (the real one)

The captured "SQUABBLE! button next to Merge/Squash/Rebase" placement is *not
possible* — GitHub Apps cannot inject buttons into the native merge box.

Primary:: Checks-API `requested_action` button ("Squabble!") in the PR checks
section — the closest native button GitHub actually exposes.
Secondary:: `/squabble` PR comment command; `squabble` label.

== v0.1 fight playbook

Mined from the 2026-06-22 rsr-template lockdown (effectively a manual squabbler
run). All moves *satisfy* the gate, never bypass it.

. *Stale required-context* → ground-truth emitted check names vs the required set;
  reconcile names so the real check satisfies the requirement.
. *Path-filter deadlock* → inject the documented job-level phantom pass-through so
  the required check legitimately completes on off-path PRs.
. *Reusable-workflow pin drift* → re-pin stale/broken SHA to current.
. *Modify/delete & rebase conflicts* → resolve, re-run.
. *Check-name ground-truthing* → shared substrate for moves 1 and 2.
. *Unsatisfiable `code_scanning` ruleset rule* (**flagship v0.1 move**) → a branch
  ruleset's `code_scanning` rule lists a tool in `code_scanning_tools` that never
  uploads a per-PR-merge-commit SARIF analysis for the repo — e.g. `Hypatia`
  (gated via a status-check context, not a code-scanning upload) or `Scorecard`
  (uploads 0 analyses estate-wide). The rule then waits forever on an analysis
  result that never arrives, so every PR sticks on "Waiting for code scanning
  results from <tool> for the commit …". Fully mechanical fix: trim
  `code_scanning_tools` down to only tools that genuinely upload SARIF per PR
  (`CodeQL`), and gate the others via `required_status_checks` contexts instead.
+
NOTE: This is a *confirmed, recurring, production* class — it deadlocked PRs on
multiple estate repos the week of 2026-06-30 (`rsr-template-repo`, `standards`,
`odds-and-sods-package-manager`, …). Already diagnosed and remediated on hypatia
itself (see hypatia issues #470 and #473); the canonical provisioner
`git-scripts/scripts/branch-protection-apply.sh` is now CodeQL-only, but stale
live `Base` rulesets persist. Because the fix is purely list-trimming with a
verifiable satisfaction condition, it is the ideal *first* squabbler move once
the tool exists.

== Outcome contract

GREEN:: Landed via a permitted (arena) strategy on a legitimately satisfied gate.
YELLOW:: Red→improved but not won; handed back to the human.
RED + report:: Could not win; structured report emitted for the human.
